## Q1: What is the difference between `conda activate ./venv` and activating by name?

**A:** The difference is how conda locates the environment:

- `conda activate ./venv` ŌĆö activates an environment stored at a specific **directory path** (created with `--prefix ./venv`). Not listed in `conda env list`. Prompt shows full path.
- `conda activate <name>` ŌĆö activates an environment stored in conda's **central location** (created with `-n name`). Visible in `conda env list`. Prompt shows short name.

---

## Q2: What are the benefits and differences of both?

**Path-based (`./venv`):**
- Environment lives inside the project folder
- Good for Docker/containers ŌĆö self-contained
- Easy to delete ŌĆö just delete the folder
- Ugly long prompt

**Named (`myenv`):**
- Clean short prompt
- Always visible in `conda env list`
- Reusable across multiple projects
- Stored centrally ŌĆö not tied to project folder

| Feature | `./venv` (path) | `myenv` (named) |
|---|---|---|
| Best for | Project-specific isolation | Reusable / shared envs |
| Prompt | Full path | Clean short name |
| Visible in `conda env list` | No | Yes |
| Docker friendly | Yes | No |

## Q6: Why do we use `-e .` in requirements.txt?

**A:** `-e .` means **editable install** of the current package.

- `.` ŌĆö current directory (project root)
- `-e` ŌĆö editable/development mode

When you run `pip install -r requirements.txt`, it tells pip to install your `networksecurity` package locally so you can import from it anywhere inside the project.

**Without `-e .`:**
```python
from networksecurity.logging import logger   # ImportError
```

**With `-e .`:**
```python
from networksecurity.logging import logger   # Works
```

| | Normal install | Editable `-e .` |
|---|---|---|
| Code changes | Must reinstall | Reflected immediately |
| Used for | Production / third-party libs | Your own package in development |
| How it works | Copies files to `site-packages` | Creates a link to your project folder |

**Note:** Requires `setup.py` to be filled in with package metadata ŌĆö currently empty in this project.

## Q7: Why do we create `setup.py`?

**A:** `setup.py` turns your project folder into a proper Python package that can be installed like any other library.

**Without `setup.py`:** Code only works if you run it from the project root folder.

**With `setup.py` + `pip install -e .`:** Your package is importable from anywhere inside the project.

```python
from networksecurity.logging import logger       # works from any file
from networksecurity.exception import exception  # works from any file
from networksecurity.utils import main_utils     # works from any file
```

**What `setup.py` tells Python:**
- `name` ŌĆö package name
- `version` ŌĆö version number
- `packages=find_packages()` ŌĆö auto-finds all subfolders with `__init__.py`
- `install_requires=get_requirements()` ŌĆö lists all dependencies

**Simple analogy:**

| Without `setup.py` | With `setup.py` |
|---|---|
| Script you can only run from one folder | Installed app usable from anywhere |
| Like needing to be in pandas' folder to use it | Your package works like pandas or numpy |

**In short:** `setup.py` + `-e .` = your own code behaves like `pandas` or `numpy` ŌĆö importable everywhere in the project.

## Q8: What is the KS Test (Kolmogorov-Smirnov Test)?

**A:** KS test checks ŌĆö *"Are these two datasets from the same distribution?"*

In this project it's used to detect **data drift** ŌĆö has the data changed between training and new incoming data?

**Simple analogy:** Model trained on January data. June data arrives. KS test asks: *"Is June's data similar enough to January's, or has it drifted?"*

**How it works:**
```python
from scipy.stats import ks_2samp
result = ks_2samp(base_df[column], current_df[column])
```

Returns 2 values:

| Value | Meaning |
|---|---|
| `result.statistic` | How different the two distributions are (0 = identical, 1 = completely different) |
| `result.pvalue` | Probability that both came from the same distribution |

**Decision rule:**
```python
threshold = 0.05
if pvalue >= threshold:
    # no drift
else:
    # drift detected ŌåÆ model may need retraining
```

| pvalue | Meaning |
|---|---|
| `0.9` | 90% chance same distribution ŌåÆ no drift |
| `0.01` | Only 1% chance same distribution ŌåÆ drift! |

**In this project:** Runs KS test on every column. If any column drifts, it's flagged in the drift report and validation fails.

**Why it matters:** If phishing URL patterns change over time (attackers use new techniques), the model becomes less accurate. KS test catches this early so the model can be retrained.
